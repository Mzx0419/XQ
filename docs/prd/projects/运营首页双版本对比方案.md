# 运营首页双版本对比方案（老板结论版 + 运营执行版）

## 1. 背景与对比对象

- 对比页面：
  - 版本A：运营偏好版（V1）
  - 版本B：老板偏好版（总览风格）
- 对比目标：在同一个首页中同时满足
  - 管理层：快速判断经营好坏与异常点
  - 运营层：快速进入高优先任务并执行
- 输出结论：推荐采用 **50/50 融合方案**，即“经营总览层 + 运营行动层 + 业务明细层（折叠）”。

## 2. 评估口径（统一给老板与运营）


| 评估维度  | 关注问题        | 评分标准（高分更优）        |
| ----- | ----------- | ----------------- |
| 首屏可读性 | 用户第一眼能否看懂重点 | 3秒内识别页面主目标与核心指标   |
| 决策效率  | 是否能快速判断经营状态 | 5秒内可回答“好/坏、异常在哪里” |
| 执行效率  | 是否能快速进入任务   | 10秒内可进入首个高优先动作    |
| 信息负荷  | 是否过多信息同屏竞争  | 首屏信息分层清晰，低频内容可折叠  |
| 扩展性   | 后续新增模块是否易接入 | 模块边界清晰，可增量插入      |


## 3. 一页结论摘要（老板结论版）

### 3.1 一句话结论

- V1 更适合“持续执行”，老板版更适合“瞬时总览”；若二选一，会牺牲另一方核心诉求。
- 最优决策不是二选一，而是融合：先让老板看得懂，再让运营做得快。

### 3.2 两版价值与短板


| 版本       | 核心价值          | 主要短板              | 对业务影响          |
| -------- | ------------- | ----------------- | -------------- |
| V1（运营偏好） | 任务路径清晰、动作入口稳定 | 总览感偏弱，管理层首屏判断成本高  | 执行效率高，管理判断慢    |
| 老板偏好版    | KPI总览强、全局感好   | 执行入口相对分散，动作引导不够直接 | 判断效率高，一线执行可能绕路 |


### 3.3 推荐方案（唯一推荐）

- 采用融合版信息架构：
  1. **经营总览层**（首屏上半）：3个关键KPI（营收、目标达成、异常数）
  2. **运营行动层**（首屏中段）：今日必须做、本周跟进
  3. **业务明细层**（首屏下段）：默认折叠，按需展开
- 推荐原因：
  - 同时覆盖“判断快”和“行动快”两类刚需
  - 不推翻现有运营习惯，培训和迁移成本更低
  - 结构可扩展，后续新增运营专题不会破坏首屏秩序

### 3.4 业务收益预估（用于决策）

- 管理视角：首屏判断时间预计降低 30%~50%
- 运营视角：首个任务到达时间预计降低 20%~35%
- 组织视角：页面理解分歧减少，跨角色沟通成本下降

## 4. 详细对比与落地方案（运营执行版）

### 4.1 模块级对比


| 模块   | V1现状         | 老板版现状      | 融合方案动作              |
| ---- | ------------ | ---------- | ------------------- |
| 顶部区  | 角色信息稳定，偏执行语境 | 总览语境更强     | 增加“总览/执行”双Tab（单激活态） |
| KPI区 | 指标有但管理摘要不足   | 指标聚合清晰、阅读快 | 固定3个核心KPI，强调数字层级    |
| 任务区  | 强动作导向        | 任务导向弱于总览导向 | 拆为“今日必须做/本周跟进”      |
| 明细区  | 信息完整但密度偏高    | 信息分布更分散    | 低频项折叠，保留摘要入口        |
| 底部导航 | 使用稳定         | 页面纵深更长     | 保持稳定，不引入额外学习成本      |


### 4.2 用户旅程对比（同角色：运营）

1. 首次打开：先看“经营总览”判断今日状态
2. 识别异常：在异常KPI直接看到“待处理”提示
3. 进入任务：进入“今日必须做”首卡执行
4. 复盘查看：需要时展开“业务明细层”查看细项

### 4.3 风险点与应对


| 风险点  | 触发场景          | 应对策略                    |
| ---- | ------------- | ----------------------- |
| 信息过载 | 首屏堆叠过多指标和卡片   | 严控首屏仅保留3个核心KPI，低频折叠     |
| 入口分散 | 总览与执行动作并存导致犹豫 | 明确分层：先总览再行动，动作区紧随KPI    |
| 状态残留 | Tab切换后样式混乱    | 强制单激活态：背景/文字/边框/阴影一次性切换 |


### 4.4 交互与视觉规范（执行约束）

- 双Tab约束（总览/执行）：
  - 同一时刻仅允许一个激活态
  - 切换时必须清空上一个激活项的所有高亮残留
- 信息层级约束：
  - 核心数字 > 环比/同比 > 说明文字
  - “今日必须做”永远位于“本周跟进”之前
- 折叠层约束：
  - 默认折叠可延后明细
  - 折叠摘要文案必须明确可点击预期

## 5. 验收清单

### 5.1 结果指标（上线后观察）


| 指标        | 目标值    | 观察窗口 | 判定方式            |
| --------- | ------ | ---- | --------------- |
| 管理判断时长    | <= 5秒  | 7天   | 可在首屏回答“好坏与异常点”  |
| 运营首任务到达时长 | <= 10秒 | 7天   | 从进入首页到点击首个高优先任务 |
| 首屏滚动深度    | <= 2屏  | 7天   | 首屏到核心行动区不超过两次滚动 |
| Tab状态正确率  | 100%   | 回归测试 | 无混合态/残留高亮       |


### 5.2 上线节奏

1. 小流量灰度（10%用户，3~5天）
2. 指标回收与问题清单复盘
3. 全量发布（满足阈值后）

### 5.3 回滚条件（保护执行效率）

- 若“首任务到达时长”显著上升（超过基线 15%），立即回滚到上个稳定版本
- 若 Tab 状态错误影响关键路径，先热修复后再扩大流量

## 6. 备选方案触发条件（非默认）

- 仅在以下情况下考虑偏向单侧方案：
  - 若管理层判断效率长期不达标：短期提高“总览层”权重
  - 若一线执行效率明显下滑：短期提高“行动层”权重
- 默认策略仍为 50/50，不建议直接回到纯V1或纯老板版。
